# Why sdd exists

> **Explanation:** Intent, tradeoffs, and what we refuse to become.

## What · So what · Now what

| | |
|--|--|
| **What** | Local Spec-Driven Development CLI that pairs with AI coding agents |
| **So what** | You get durable change packs without cloud SDD or IDE lock-in |
| **Now what** | Use the [first-change tutorial](/tutorials/first-change) once, then customize only if needed |

## PREP

**Point:** Agents need a process spine; humans need it light enough to keep.

**Reason:** Unstructured chat loses decisions; enterprise gates block solos.

**Example:** Spec Kit–like init (`--ai grok|copilot|claude`) + YAML stages + markdown artifacts.

**Point:** `sdd` is the spine; the agent is the writer.

## Problem · Solution · Benefit

| | |
|--|--|
| **Problem** | Vibe coding ships fast and forgets *why*. Heavy SDD ships slowly and frustrates. |
| **Solution** | Progressive structure: hotfix 3 stages, feature more, enterprise optional ARB — all local. |
| **Benefit** | Same tool from solo weekend to team PR; agents read one protocol + live context. |

## What we are not

- Not a required IDE extension  
- Not a hosted collaboration suite  
- Not “skills packs” replacing a single playbook  
- Not mandatory archive/evidence folders (opt-in only)
